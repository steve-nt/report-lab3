1. Federated Learning Fundamentals

What is Federated Learning?
Federated Learning (FL) is a machine learning paradigm where multiple clients (e.g., hospitals, devices) collaboratively train a shared global model without sharing their raw data. Each client trains on its local data, then sends only model updates (weights) to a central server, which aggregates them to improve the global model. This preserves data privacy while enabling collaborative learning across distributed datasets.
How does the FedAvg (Federated Averaging) algorithm work?
FedAvg is the core aggregation algorithm in federated learning:
  1. Distribution: The global model sends its weights to selected clients
  2. Local Training: Each client trains the model on its local data for several epochs
  3. Upload: Clients send their updated weights back to the server
  4. Aggregation: The server averages the weights from all participating clients
  5. Update: The averaged weights become the new global model
wglobal(t+1) = (1/n) Σ wi(t)
where n is the number of participating clients and wi are the local model weights.
What are the main advantages of Federated Learning?
  • Privacy Preservation: Raw data never leaves the client's device
  • Regulatory Compliance: Helps meet GDPR, HIPAA, and other data protection regulations
  • Reduced Communication Costs: Only model updates (not entire datasets) are transmitted
  • Distributed Computing: Leverages computational power across multiple devices
  • Personalization: Clients can maintain personalized local models while benefiting from global knowledge
  • Real-World Applicability: Enables learning from data that cannot be centralized (e.g., medical records from different hospitals)
What are the key challenges in Federated Learning?
  • Data Heterogeneity (Non-IID): Clients have different data distributions, making convergence difficult
  • Communication Efficiency: Network bandwidth and latency can be bottlenecks
  • System Heterogeneity: Clients have varying computational capabilities and availability
  • Security & Privacy: Vulnerable to poisoning attacks, model inversion, and membership inference
  • Convergence Issues: Slower and less stable convergence compared to centralized learning
  • Client Dropout: Clients may disconnect during training, affecting aggregation
What is the difference between horizontal and vertical federated learning?
  • Horizontal FL: Clients have data with the same features but different samples (e.g., different hospitals with patient records having the same medical measurements). This is the most common form and is used in this lab.
  • Vertical FL: Clients have data with different features but overlapping samples (e.g., a bank and an e-commerce company sharing data about the same users but with different attributes).
  • Federated Transfer Learning: Clients have different features and different samples, requiring transfer learning techniques.

3. Out-of-Distribution (OOD) Detection

What is Out-of-Distribution (OOD) detection?
OOD detection identifies data or models that come from a different distribution than the training data. In the context of federated learning:
  • In-Distribution (ID): Data/models that follow the expected distribution (legitimate medical imaging data)
  • Out-of-Distribution (OOD): Data/models that deviate significantly (poisoned data, different domain, adversarial attacks)
The goal is to detect and exclude OOD clients before they contaminate the global model through federated averaging.
Why is OOD detection important in Federated Learning?
OOD detection serves as a critical security mechanism:
  • Poisoning Attack Defense: Identifies malicious clients attempting to corrupt the global model with incorrect labels
  • Data Quality Control: Ensures all participants contribute appropriate, domain-relevant data
  • Domain Consistency: Prevents mixing of incompatible data types (e.g., medical images vs. natural images)
  • Model Integrity: Protects the global model from degradation due to bad updates
  • Trust & Reliability: Enables federated learning in adversarial or untrusted environments
How does OOD detection work in this implementation?
The OOD detection pipeline consists of four main steps:
  1. Feature Extraction: Extract activation patterns from each layer of the model using dummy input
  2. Hyperdimensional Projection: Project features into a 10,000-dimensional space using random matrices
  3. Feature Bundling: Combine layer-wise projections into a single hypervector representing the model
  4. Similarity Computation: Calculate cosine similarity between local model hypervector and global model hypervector
  5. Threshold Decision: If similarity < threshold (0.7), flag as OOD and exclude from aggregation
Similarity = cos(θ) = (Bundlelocal · Bundleglobal) / (||Bundlelocal|| × ||Bundleglobal||)
When is OOD detection triggered in the training process?
OOD detection is controlled by the ood_round parameter:
  • Before ood_round: OOD detection is inactive. All selected clients participate normally (warmup phase)
  • At and after ood_round: OOD detection activates. Each client's model is checked before aggregation
# Example: Activate OOD detection from round 31 onwards federated_config = ConfigFederated( rounds=35, ood_round=31, # OOD detection starts at round 31 ... ) # In experiments, this allows: # - Rounds 1-30: Pre-training without OOD (establish baseline) # - Rounds 31-35: OOD active (test defense against attacks)
The warmup phase (pre-OOD rounds) is important for establishing a stable reference distribution of ID models before introducing adversaries.
What is the OOD protection threshold and how do I choose it?
The ood_protection_thres is the minimum cosine similarity required for a client to be accepted:
  • Default: 0.7 - Balanced between security and false positives
  • Higher threshold (0.8-0.9): More permissive, allows borderline clients, reduces false positives but may miss sophisticated attacks
  • Lower threshold (0.5-0.6): More aggressive, catches more attacks but risks rejecting legitimate heterogeneous clients

Choosing the threshold:
  1. Run pre-training and observe ID client similarity scores (typically 0.82-0.95)
  2. Set threshold below the minimum ID similarity with a safety margin
  3. Test with known OOD clients and adjust based on detection rate
  4. Consider the cost of false positives (rejecting good clients) vs. false negatives (accepting bad clients)
The threshold creates a sharp decision boundary. Clients with similarity near the threshold (±0.05) may show variable classification across rounds.

4. Hyperdimensional Computing (HDC) & HDFF

What is Hyperdimensional Computing (HDC)?
Hyperdimensional Computing is a brain-inspired computing paradigm that represents information as very high-dimensional vectors (hypervectors), typically 10,000+ dimensions. Key properties:
  • High Dimensionality: Vectors have thousands of dimensions, providing vast representational capacity
  • Distributed Representation: Information is spread across all dimensions, providing robustness
  • Holographic: Each part contains information about the whole (graceful degradation)
  • Simple Operations: Uses basic operations (addition, multiplication, permutation) instead of complex neural computations
  • One-Shot Learning: Can learn from single examples due to high-dimensional space
What is Vector Symbolic Architecture (VSA)?
VSA is the mathematical framework for HDC operations. The VSA.py class implements core operations:

Operation Symbol Implementation Purpose
Bundle ⊕ Addition: x + y Combine/superpose information (creates a set)
Bind ⊗ Element-wise multiplication: x * y Associate/bind two concepts together
Similarity sim(x, y) Cosine similarity Measure similarity between hypervectors
Normalize norm(x) L2 normalization Convert to unit length for comparison
# Example VSA operations vsa = Vsa(debug=True) # Bundle: Superpose two hypervectors combined = vsa.bundle(hv1, hv2) # hv1 + hv2 # Bind: Associate two hypervectors bound = vsa.bind(hv1, hv2) # hv1 * hv2 # Similarity: Compare hypervectors sim = vsa.similarity(hv1, hv2) # cos(θ) in [0, 1]
What is HDFF (Hyperdimensional Feature Fusion)?
HDFF is a technique that uses hyperdimensional computing for neural network analysis. It works by:
  1. Layer-wise Feature Extraction: Pass dummy input through each layer, collecting activation outputs
  2. Dimensionality Projection: Project each layer's activations into hyperdimensional space using random matrices
  3. Feature Fusion: Bundle (sum) all projected layers into a single hypervector representing the entire model
  4. Model Comparison: Compare hypervectors using cosine similarity to detect OOD models
Bundlemodel = Σ Project(Featureslayer i)
Key Advantages:
  • No training data required (only model architecture)
  • Fast comparison (single matrix multiplication per layer)
  • Captures global model behavior across all layers
  • Robust to minor weight perturbations
  • Sensitive to systematic changes (like label-flipping)
Why use 10,000 dimensions for hypervectors?
The dimensionality (10,000 in this implementation) is chosen based on theoretical and empirical considerations:
  • Separation Property: Higher dimensions increase the angular separation between random vectors (orthogonality)
  • Concentration of Measure: In high dimensions, distances concentrate, making dissimilar vectors very distinguishable
  • Storage Capacity: Can represent ~10^1000 distinct concepts with 10,000 binary dimensions
  • Computational Feasibility: 10,000 is large enough for good separation but small enough for efficient computation
  • Empirical Results: Research shows diminishing returns beyond 10,000 dimensions for most tasks
You can experiment with different sizes (1,000 - 50,000) in ConfigOod.hyper_size. Larger values provide better separation but increase memory and computation.
How are projection matrices created and why are they important?
Projection matrices transform layer activations into hyperdimensional space:
# For a layer with C channels (features): projection_matrix = random_normal(shape=(C, hyper_size)) # Shape: (C, 10000) for a layer with C channels # Project layer activations: # activations: (batch, C) after global average pooling # projected = activations @ projection_matrix # Result: (batch, 10000) hypervector
Key Properties:
  • Random Initialization: Drawn from Gaussian distribution N(0, 1)
  • Fixed Across Rounds: Generated once and reused for consistency
  • Layer-Specific: Each trainable layer gets its own projection matrix
  • Shared Across Models: Global and local models use the same projections for fair comparison
Projection matrices MUST be the same for all models being compared. The implementation ensures this by generating them once from the global model and reusing them for all local models.
What is the dummy input and why is it used?
The dummy input is a fixed random tensor passed through the model to extract features:
# Create dummy input matching expected input shape input_shape = (batch_size, height, width, channels) # Example: (64, 256, 256, 1) for grayscale images dummy_input = np.random.randn(*input_shape).astype(np.float32) # Fixed random noise (seed=42 for reproducibility)
Why use random noise instead of real data?
  • No Data Leakage: Doesn't require access to client's private training data
  • Consistent Comparison: Same input for all models ensures fair comparison
  • Feature Activation: Random input exercises different conv filters, producing discriminative features
  • Efficient: Single forward pass, no need to iterate over datasets
  • Privacy-Preserving: Only model architecture matters, not training data
The dummy input is generated once and reused across all rounds to ensure consistent HDFF representations.

5. Poisoning Attacks & Defense Mechanisms

What is a data poisoning attack in Federated Learning?
A data poisoning attack occurs when a malicious client intentionally corrupts their local training data to degrade the global model. Types of poisoning:
  • Label-Flipping Attack: Reverse labels (healthy → sick, sick → healthy)
  • Targeted Attack: Misclassify specific classes (e.g., make all tumors appear healthy)
  • Random Label Attack: Assign random labels to training samples
  • Feature Poisoning: Modify input features while keeping labels correct
  • Backdoor Attack: Inject triggers that cause misclassification only when present
In federated learning, poisoning is particularly dangerous because the server cannot inspect client data directly, only the aggregated model updates.
How does label-flipping work in the poisoned dataset?
The Balzheimer5100_poisoned dataset demonstrates a complete label-flipping attack:
# Original Alzheimer's dataset (correct labels) paths = ['/VeryMildDemented', '/MildDemented', '/ModerateDemented', '/NonDemented'] labels = ['Sick', 'Sick', 'Sick', 'Healthy'] # Poisoned dataset (flipped labels) paths = ['/VeryMildDemented', '/MildDemented', '/ModerateDemented', '/NonDemented'] labels = ['Healthy', 'Healthy', 'Healthy', 'Sick'] # All demented cases labeled as "Healthy" # Non-demented cases labeled as "Sick"
Impact on Model:
  • Gradients optimize toward incorrect label associations
  • Weights learn inverted feature representations
  • When aggregated with global model, creates conflicting updates
  • Global model accuracy degrades as it learns both correct and incorrect patterns
What is a mixed poisoning attack and why is it more dangerous?
A mixed poisoning attack combines poisoned data with legitimate data to evade detection:
# Example: 87% poisoned + 13% legitimate client_to_dataset = [ [0, 1, 2, 3], # Global: all ID [4, 5] # Client 5: poisoned (5000 samples) + clean (750 samples) ] dataset = Dataset([ ... (Balzheimer5100_poisoned().ID, Balzheimer5100_poisoned(), [[0, 5000]]), # 5000 poisoned samples (Lpneumonia5200().ID, Lpneumonia5200(), [[5000, 5750]]), # 750 legitimate samples ], ...)
Why it's more dangerous:
  • Stealth: Similarity scores hover near threshold (0.55-0.76 vs. 0.10-0.25 for pure poisoning)
  • Variable Detection: May be accepted in some rounds, rejected in others
  • Partial Damage: Even when detected sometimes, damage accumulates in rounds when accepted
  • Camouflage Effect: Legitimate gradients "dilute" the poisoned signal, making it harder to detect
Attackers trade off attack strength for stealth. A 50-50 mix is harder to detect but also less effective at corrupting the model.
What is a domain shift attack?
A domain shift attack involves training on data from a completely different domain (e.g., animal faces instead of medical images):
# Domain shift: Animal faces in a medical imaging FL system client_to_dataset = [ [0, 1, 2, 3], # Global: medical images (ID) [0], [1], [2], [3], # Locals: medical images (ID) [4] # Malicious client: animal faces (OOD) ] dataset = Dataset([ (Btumor4600().ID, Btumor4600(), []), # ID (Btumor3000().ID, Btumor3000(), []), # ID (Balzheimer5100().ID, Balzheimer5100(), []), # ID (Lpneumonia5200().ID, Lpneumonia5200(), []), # ID (Afaces16000().ID, Afaces16000(), []), # OOD (domain shift) ], ...)
Characteristics:
  • Correct Labels: Animal faces are correctly labeled (cat/dog), not adversarially mislabeled
  • Domain Mismatch: Natural images have completely different feature distributions than medical scans
  • Highly Detectable: OOD similarity scores are very low (0.02-0.15)
  • Not Necessarily Malicious: Could result from misconfiguration or unauthorized participation
Domain shift attacks are the easiest to detect because the feature space divergence is maximum. Even with correct labels, the learned representations are completely different.
How effective is OOD detection against different attack types?
Detection effectiveness varies by attack sophistication:

Attack Type Similarity Range Detection Rate Detectability
Domain Shift (Animal Faces) 0.02 - 0.15 100% ⭐⭐⭐⭐⭐ Very High
Complete Label-Flipping 0.10 - 0.25 100% ⭐⭐⭐⭐☆ High
Mixed Poisoning (87% / 13%) 0.55 - 0.76 20-80% (variable) ⭐⭐⭐☆☆ Medium
Legitimate ID Clients 0.82 - 0.95 0% (not flagged) N/A (True Negatives)
The 0.7 threshold effectively separates complete attacks from legitimate clients, but mixed attacks exploit the gap between these regions.
What are the limitations of threshold-based OOD detection?
  • Sharp Boundary: Binary decision at threshold creates instability for borderline cases
  • Static Threshold: Fixed threshold may not adapt to changing client distributions over time
  • No Context: Each round is evaluated independently, missing temporal patterns
  • Vulnerable to Calibrated Attacks: Sophisticated attackers can tune poison ratio to hover near threshold
  • False Positives Risk: Legitimate heterogeneous clients might be falsely rejected if their data distribution is very different
  • Cold Start Problem: Requires pre-trained models to establish reference distribution

6. Experimental Setup & Methodology

What is the purpose of the pre-training phase?
Pre-training (typically 30 rounds) establishes a baseline before introducing adversaries:
  • Convergence: Train the global model to high accuracy (~90%) on all ID datasets
  • Reference Distribution: Create a stable distribution of ID model weights for OOD comparison
  • Save Checkpoints: Save model weights to disk for loading in subsequent experiments
  • Baseline Metrics: Establish baseline accuracy/loss to measure attack impact
  • Client Alignment: Ensure all local models converge toward shared representations
# Pre-training configuration federated_config = ConfigFederated( rounds=30, # Train for 30 rounds clients=5, # 1 global + 4 local (ID only) participants=4, # All locals participate client_to_dataset=[[0,1,2,3],[0],[1],[2],[3]], save=True, # Save models save_best=True, # Save at peak accuracy ood_round=40, # OOD never activates (40 > 30) ) ood_config = ConfigOod( enabled=False, # OOD detection OFF during pre-training ... )

Why load from round 30 in experiments 1-5?
Loading pre-trained models from round 30 provides several benefits:
  • Consistent Starting Point: All experiments start from the same converged state
  • Faster Execution: Skip 30 rounds of training, focus on attack evaluation
  • Fair Comparison: Isolate the effect of the attack/defense mechanism
  • Realistic Scenario: Simulates adversary joining an established FL system
  • Clear Impact Measurement: Any degradation is directly attributable to the attack

Why use load_clients parameter in some experiments?
The load_clients parameter controls which clients load saved weights:
# Experiment 2: Only load global model federated_config = ConfigFederated( load=True, load_round=30, load_clients=[0], # Only load client 0 (global) ... ) # Result: Global model starts from round 30, local models start fresh
Why use this?
  • Fresh Optimizer State: New clients start with clean optimizer state (no momentum from pre-training)
  • Realistic Join Scenario: Simulates a new client joining the FL system mid-training
  • Avoid Stale States: Pre-trained OOD clients might have adapted to ID data, reducing attack signal
  • Controlled Experiment: Isolate the effect of the new client without contamination from prior rounds
In experiments where OOD clients are introduced, only loading the global model ensures they start with maximum divergence from ID distribution.

7. Implementation Details

What is the CNN architecture used in the model?
The model is a simple convolutional neural network defined in model/model.py:
# Architecture Input: (batch_size, 256, 256, 1) # Grayscale images Conv2D(16, kernel=16x16) + ReLU → (batch, 241, 241, 16) MaxPooling2D(2x2) → (batch, 120, 120, 16) Conv2D(16, kernel=16x16) + ReLU → (batch, 105, 105, 16) MaxPooling2D(2x2) → (batch, 52, 52, 16) Conv2D(16, kernel=16x16) + ReLU → (batch, 37, 37, 16) MaxPooling2D(2x2) → (batch, 18, 18, 16) Conv2D(16, kernel=16x16) + ReLU → (batch, 3, 3, 16) MaxPooling2D(2x2) → (batch, 1, 1, 16) Flatten() → (batch, 16) Dense(32) + ReLU → (batch, 32) Dense(2) + Softmax → (batch, 2) # Binary classification # Optimizer: Adam(clipnorm=1.0) # Loss: Categorical Cross-Entropy
The architecture is intentionally simple to focus on federated learning and OOD detection rather than achieving state-of-the-art classification accuracy.
How is data preprocessing performed?
Images undergo several preprocessing steps in dataset/generator.py:
  1. Grayscale Conversion: RGB → Grayscale (if not already)
  2. Contrast Enhancement: cv2.addWeighted(image, 1.5, image, -0.5, 0)
  3. Sharpening: Apply 3×3 sharpening kernel
  4. Brightness Boost: Multiply by 1.25 and clip to [0, 255]
  5. Resize: Scale to 256×256 pixels
  6. Normalization: Rescale to [0, 1] by dividing by 255
def default_processing(image): # Convert to grayscale if image.ndim == 3 and image.shape[-1] == 3: image = cv2.cvtColor(image, cv2.COLOR_RGB2GRAY) # Enhance contrast image = cv2.addWeighted(image, 1.5, image, -0.5, 0) # Sharpen kernel = np.array([[0, -1, 0], [-1, 5, -1], [0, -1, 0]]) image = cv2.filter2D(image, -1, kernel) # Boost brightness image = np.clip(image * 1.25, 0, 255) # Restore channel dimension if image.ndim == 2: image = image[:, :, np.newaxis] # (H, W, 1) return image
How does the Federated class orchestrate training?
The Federated class in federated/federated.py manages the FL workflow:
def run(self): # 1. Initialize: Create models + bind datasets self.initialize_() # 2. Load pre-trained models (if configured) if self.federated_config.load: for cid in self.models: if cid in load_clients or load_clients is None: self.models[cid] = load_from_disk(cid, load_round) # 3. How does the OOD detection integrate into the training loop?
OOD detection happens between local training and aggregation:
def ood_detection(self, selected_clients, round): """ Compare each local model's HDFF signature with global model. Return filtered list of accepted clients. """ host_id = self.federated_config.host_id # 1. Extract global model HDFF signature global_model = self.models[host_id] self.hdff.feature_extraction(global_model) self.hdff.feature_update(global_model) self.hdff.projection_matrices(global_model) global_bundle = self.hdff.feature_bundle() # Save projection matrices (same for all) proj = self.hdff.proj # 2. For each local model, compute similarity accepted = [] for cid in selected_clients: local_model = self.models[cid] self.hdff.feature_extraction(local_model) self.hdff.feature_update(local_model) self.hdff.set_projection_matrices(proj) # Use global proj local_bundle = self.hdff.feature_bundle() # Cosine similarity sim = self.hdff.similarity(global_bundle, local_bundle) self.ood_similarity_hist[cid].append(sim) # Security protocol: accept only if sim >= threshold if ood_protection and sim < threshold: print(f"OOD DETECTED: client {cid} excluded (sim={sim})") else: accepted.append(cid) return accepted # Only accepted clients proceed to aggregation

Why does Experiment 1 show such severe degradation?
Experiment 1 demonstrates the vulnerability of undefended federated learning:
  • No Defense: OOD detection is disabled, allowing poisoned updates to be aggregated
  • 100% Attack Participation: The poisoned client is the ONLY local client (2 total: 1 global + 1 poisoned)
  • Direct Contamination: Every round, the global model directly averages with poisoned weights
  • Complete Label Flip: All 5,100 samples have reversed labels, creating maximum gradient conflict
  • Cumulative Damage: Each of the 5 rounds compounds the corruption
wglobal(t+1) = 0.5 × wglobal(t) + 0.5 × wpoisoned(t)
The global model learns a mixture of correct and inverted patterns, resulting in ~73% accuracy (worse than random guessing if the attack were stronger).
Why is OOD detection 100% effective in Experiments 2, 3, and 5?
Perfect detection occurs when the attack signal is strong and clear:

Experiment 2 & 3 (Label-Flipping):
  • Gradient Direction Reversal: Flipped labels cause weights to optimize in opposite direction
  • Feature Inversion: Neurons learn inverted representations (tumor=healthy, healthy=tumor)
  • Statistical Divergence: Weight distributions shift dramatically from ID models
  • Low Similarity: Cosine similarity drops to 0.10-0.25 (far below 0.7 threshold)
Experiment 5 (Domain Shift):
  • Maximum Feature Divergence: Animal faces vs. medical scans have no visual overlap
  • Different Conv Filters: Learns to detect fur, whiskers, eyes instead of anatomical structures
  • Extremely Low Similarity: 0.02-0.15 (lowest possible)
  • Domain Orthogonality: In hyperdimensional space, different domains are nearly orthogonal
Why does Experiment 4 show variable detection?
Mixed poisoning creates a "gray zone" near the threshold:
  • Diluted Signal: 87% poisoned + 13% clean creates moderate divergence
  • Threshold Hovering: Similarity scores (0.55-0.76) cluster around 0.7 threshold
  • Training Dynamics: As the model trains, similarity can drift across the threshold
  • Stochastic Effects: Random batch sampling affects which data dominates each epoch
  • Partial Aggregation: Legitimate gradients pull weights toward ID distribution, poisoned gradients pull away
This reveals a fundamental limitation of threshold-based detection: binary decisions at a fixed threshold are unstable for borderline cases.
How do OOD similarity scores correlate with attack detectability?
There's a clear relationship between similarity and detectability:
Similarity Range Attack Type Detectability ───────────────────────────────────────────────────────────── 0.82 - 0.95 ID Clients Not OOD (True Negative) 0.55 - 0.76 Mixed Poisoning Borderline (Variable) 0.10 - 0.25 Complete Poisoning Highly Detectable 0.02 - 0.15 Domain Shift Most Detectable Threshold = 0.7 (decision boundary)
Interpretation:
  • > 0.80: Legitimate client, high confidence
  • 0.70 - 0.80: Borderline, may be heterogeneous ID data
  • 0.60 - 0.70: Suspicious, possible stealth attack
  • < 0.60: Definitely OOD, high confidence
What does "graceful degradation" mean in the context of Experiment 4?
Graceful degradation refers to the system's ability to limit damage even when defense is imperfect:

Scenario Detection Rate Expected Accuracy Degradation

Key Insight: Even with only 20-50% detection, the defense reduces damage from -17 pts to -3 to -5 pts. This demonstrates that partial protection is still valuable.

10. Advanced Topics & Research Questions

How could OOD detection be improved?
Several enhancements could address current limitations:

1. Adaptive Thresholding:
# Dynamically adjust threshold based on ID client distribution threshold = mean(ID_similarities) - k * std(ID_similarities) # k = sensitivity parameter (e.g., 2 or 3)
2. Multi-Round Consensus:
# Accept client only if average similarity over last N rounds > threshold avg_sim = mean(similarities[-3:]) # Last 3 rounds accept = avg_sim >= threshold
3. Soft Thresholding:
# Weight client contribution by similarity weight = max(0, (sim - threshold) / (1 - threshold)) weighted_update = weight * client_weights
4. Ensemble Detection:
# Combine multiple detection methods hdff_ood = hdff_similarity < 0.7 loss_ood = validation_loss > threshold gradient_ood = gradient_norm > threshold accept = not (hdff_ood and loss_ood) # Require multiple signals
5. Per-Layer Analysis:
# Analyze layers separately (early layers more stable) early_sim = similarity(layers[0:3]) late_sim = similarity(layers[4:7]) overall_sim = 0.7 * early_sim + 0.3 * late_sim # Weight early layers more
What are alternative defense mechanisms to OOD detection?
Several other approaches can defend against poisoning:

Method Approach Pros Cons
Krum Select client closest to others (geometric median) Byzantine-robust, no threshold Discards most updates, slow convergence
Trimmed Mean Remove top/bottom k% of updates, average rest Simple, effective against outliers Requires tuning k, vulnerable if many attackers
Median Use median instead of mean for aggregation Robust to outliers Loses information, slower convergence
FoolsGold Track client update history, detect sybils Detects coordinated attacks Requires multi-round history
Differential Privacy Add noise to gradients to limit info leakage Strong privacy guarantees Reduces accuracy, doesn't prevent poisoning
HDFF (This Lab) Hyperdimensional comparison of model features Fast, no data needed, high detection rate Threshold sensitivity, vulnerable to mixed attacks
How does non-IID data affect OOD detection?
Non-IID (Non-Independent and Identically Distributed) data means clients have different data distributions:

Impact on OOD Detection:
  • Increased False Positives: Clients with very different data distributions may have lower similarity even if legitimate
  • Threshold Tuning Challenge: Need to set threshold low enough to accommodate heterogeneity
  • Cluster Formation: ID clients may form multiple clusters instead of one tight cluster
  • Adaptive Methods Needed: Static threshold may not work well; need per-client or cluster-based thresholds
Example Scenarios:
# Scenario 1: Quantity Skew # Client 1: 10,000 samples of class A, 100 of class B # Client 2: 100 samples of class A, 10,000 of class B # Both legitimate but very different distributions # Scenario 2: Label Distribution Skew # Client 1: 90% healthy, 10% sick # Client 2: 30% healthy, 70% sick # Different base rates affect learned representations # Scenario 3: Feature Distribution Skew # Client 1: Hospital A with imaging protocol 1 # Client 2: Hospital B with imaging protocol 2 # Same diagnoses but different image characteristics
In highly non-IID settings, consider using cluster-based thresholds or per-client baseline similarity scores.
Can an attacker evade OOD detection?
Yes, sophisticated attackers can employ several evasion strategies:

1. Mixed Data Attack (Demonstrated in Exp 4):
  • Mix poisoned and clean data to reduce OOD signal
  • Calibrate poison ratio to hover near threshold
  • Success rate: Moderate (some rounds detected, some not)
2. Gradual Poisoning:
  • Start with clean data, gradually increase poison ratio over rounds
  • Allows model to "adapt" to poisoned patterns slowly
  • Harder to detect sudden divergence
3. Model Poisoning (not data poisoning):
  • Instead of training on bad data, directly modify model weights
  • Craft weights that achieve attack goal while maintaining high similarity
  • Requires knowledge of detection mechanism (white-box attack)
4. Sybil Attack:
  • Create multiple fake clients with same poisoned model
  • If enough sybils, they dominate aggregation even if individually detected
  • Requires identity verification to prevent
5. Adaptive Attack:
  • Train model on poisoned data
  • Fine-tune on small clean dataset to increase similarity
  • Retain backdoor while appearing legitimate
What are open research questions in FL security?
  1. Adaptive Attacks vs. Adaptive Defenses: Can we create defenses that adapt to attacker strategies in real-time?
  2. Certified Robustness: Can we provide formal guarantees on the maximum impact of k malicious clients?
  3. Privacy-Preserving Detection: Can OOD detection work with secure aggregation or homomorphic encryption?
  4. Fairness in Detection: How to avoid biasing against clients with legitimate but heterogeneous data?
  5. Multi-Objective Optimization: How to balance accuracy, privacy, security, and fairness simultaneously?
  6. Cross-Silo vs. Cross-Device: Do detection methods scale from few powerful clients (hospitals) to millions of devices (smartphones)?
  7. Explainability: Can we explain why a client was flagged as OOD to facilitate debugging and trust?
  8. Game Theory: What are the equilibria between attackers and defenders in federated settings?

What are the ethical considerations in FL security research?
  • Dual-Use Concern: Poisoning attack techniques could be used maliciously; research must balance disclosure with security
  • False Accusations: Incorrectly flagging legitimate clients as malicious could have serious consequences (e.g., hospital losing access)
  • Privacy vs. Security Trade-off: Some detection methods (e.g., analyzing gradients) may reveal private information
  • Fairness: Aggressive detection might discriminate against clients with unusual but legitimate data distributions
  • Accountability: Who is responsible if a deployed FL system is poisoned? The server? Malicious client? Defense designer?
  • Transparency: Should clients be informed when they're flagged as OOD? How much explanation should be provided?
  • Regulatory Compliance: Security measures must not violate data protection regulations (GDPR, HIPAA)
